## Minutes

- [Tong Li (Deactivated)](https://lf-hyperledger.atlassian.net/wiki/people/712020:7579aadb-a578-4296-b576-84509b88eb92?ref=confluence) suggest to isolate the authentication and authorization part, i.e., let nginx or other web proxy to help do authentication. While [hightall](https://lf-hyperledger.atlassian.net/wiki/people/70121:e9c4e0e0-079d-423a-b406-d1bcab2e0194?ref=confluence) think the API server can handle both. The existing solution is not that flexible, but can be taken as the first milestone.
- [Peng Du](https://lf-hyperledger.atlassian.net/wiki/people/712020:40cfa3db-3ae0-4442-b843-16a107ce7b9f?ref=confluence) [Luke Chan](https://lf-hyperledger.atlassian.net/wiki/people/712020:41573b41-33dc-492c-836d-536b50e9eb2a?ref=confluence) and [Mingming Tian](https://lf-hyperledger.atlassian.net/wiki/people/712020:b46cc809-7723-4103-bdce-1c3d3d41e778?ref=confluence) proposed their thoughts, too.
- [Tong Li (Deactivated)](https://lf-hyperledger.atlassian.net/wiki/people/712020:7579aadb-a578-4296-b576-84509b88eb92?ref=confluence) suggest let each agent run inside container:
  
  - Pros: No need to use python to write the agent code, any language can be taken, even with command line tool.
  - Cons: May introduce latency during agent call, and may need to consider how to secure the container call, and let the container has credential when start up.
  - Question TBD: the container will be server-less or work as daemon?
